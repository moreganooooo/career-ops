# Mode: coverletter — Cover Letter Generator

Use this mode when the user explicitly wants a cover letter, or when an application clearly requests one.

A cover letter is not automatic. Only generate one when it will genuinely help.

This mode should produce a one-page, company-specific, role-specific letter that sounds like Morgan: warm, specific, intelligent, human, and impossible to mistake for a template.

---

## When to Use

Run this mode when:
- the user asks for a cover letter
- an application explicitly requests one
- the role is strong enough to justify custom effort

Do not generate a cover letter by default for every application.

### When NOT to use

Do not generate a full cover letter when:
- the user only needs short-answer application responses
- the job is a weak or marginal fit
- the company only offers a tiny text box better handled by `apply`
- the role does not justify the time investment

In those cases, suggest a lighter response instead.

## Phase 2 — Research Before Writing

Do not write a single sentence until the company and role have been researched.

### Required research

1. Read the job description closely
2. Identify:
   - company name
   - role title
   - core responsibilities
   - what they are actually hiring this person to solve
   - tone of the posting
   - any obvious risks or title mismatches
3. Review the company’s:
   - About page
   - mission / values page
   - careers page
   - product or audience page if relevant
4. Check for an existing evaluation report in `reports/`
5. Identify 2–3 proof points from Morgan’s background that map directly to this role

### Research goals

Before drafting, answer these silently:

- Why this company specifically?
- Why this role specifically?
- What would make Morgan believable here?
- What concern might a hiring manager have at first glance?
- What is the most persuasive way to address that concern without sounding defensive?

If those answers are fuzzy, research more.

## Phase 3 — Framing Rules

Every cover letter should be built on the correct frame.

### Core rules

- Do not retell the resume in paragraphs
- Do not exaggerate titles or ownership
- Do not apologize for a nontraditional path
- Do not write generic admiration fluff
- Do not use the letter to compensate for a weak role match

### Morgan-specific framing

Use `_profile.md` as the reasoning layer.

Key standing rules:
- demonstrated function matters more than title lineage
- “outbound sequences” can be framed as lifecycle / email campaign work when appropriate
- sales-floor experience should be reframed toward writing, systems, enablement, and CRM fluency when relevant
- the long-term gap should be framed calmly and positively only if it needs to be addressed
- if the role is slightly less senior, frame interest as intentional fit, not fallback desperation

### Archetype-specific lead

If the role is:
- **Email / lifecycle**: lead with campaign systems, segmentation, performance thinking, CRM/ESP fluency
- **Enablement**: lead with content library building, training, playbooks, adoption, cross-functional alignment
- **Content / copy**: lead with writing craft, agency/editorial foundation, voice, messaging quality
- **Marketing ops**: lead with CRM hygiene, QA, process design, documentation, reporting
- **Generalist**: lead with range, adaptability, and practical execution without sounding diffuse

## Phase 4 — Writing Rules

### Rule 1 — The opening must prove this is not a template

The first sentence cannot be generic.

Bad:
- “I’m writing to apply for…”
- “I am excited to submit my application…”
- “I’ve long admired your company…”

Good openings usually do one of these:
- begin with a specific insight about the company, audience, or role
- begin with a real point of resonance tied to the mission
- begin with a sharp observation that only fits this company
- begin with a line that shows cultural fluency without sounding gimmicky

### Rule 2 — Specific beats impressive

Use real examples with stakes and texture.

Better:
- what was built
- why it mattered
- what made it difficult
- what changed because of it

Worse:
- inflated abstractions
- generic “cross-functional leader” language
- copied bullet points from the resume

### Rule 3 — The letter should sound like a person

First person is appropriate here.

The voice should be:
- warm
- precise
- self-aware
- confident
- grounded

Not:
- stiff
- overeager
- generic
- overly polished
- theatrically “passionate”

### Rule 4 — Match the company’s register

Mirror the company’s tone without mimicking their copy.

Examples:
- mission-driven org → warmer, more resonant
- playful startup → sharper, slightly more personality
- conventional B2B SaaS → measured, crisp, lightly distinctive
- advocacy / impact org → purposeful, human, values-aware

### Rule 5 — Avoid banned language

Do not use:
- passionate
- driven
- results-oriented
- dynamic
- synergy
- impactful
- proactive
- dedicated
- team player
- excited to join
- I would be a great fit
- thank you for your consideration

Also avoid obvious AI tells and overused corporate sludge.

### Rule 6 — No em dashes

Do not use em dashes in cover letters.

Use instead:
- commas
- parentheses
- colons
- short follow-up sentences

### Rule 7 — The closing should assume conversation, not beg for it

End forward, not pleading.

Bad:
- “I hope to hear from you soon.”
- “Thank you for your time and consideration.”

Better:
- a warm, direct line that points naturally toward a conversation
- a sentence connecting Morgan’s experience to the team’s next challenge
- a closing that feels grounded and open, not needy

### Rule 8 — Sign-off should follow `writing-style.md`

Do not hard-code a single sign-off for every company.

Default to a warm, professional sign-off that fits the tone. In most cases, `Warmest regards,` is still a safe default, but use `writing-style.md` as the source of truth.

## Phase 5 — Recommended Structure

Target length: **320–400 words**
Absolute maximum: **one page**

### Paragraph 1 — Hook
2–3 sentences

- open with something specific
- make clear why this role/company caught Morgan’s attention
- establish fit without listing qualifications

### Paragraph 2 — Proof
3–4 sentences

- go deeper on 1–2 relevant accomplishments
- include context, stakes, and at least one real result when appropriate
- do not dump the full resume here

### Paragraph 3 — Company Connection
2–3 sentences

- connect Morgan’s background to something specific about the company, audience, or mission
- this is where trust is earned
- avoid fake flattery

### Paragraph 4 — Close
2–3 sentences

- make the case for conversation
- keep it warm and confident
- do not repeat the entire argument

### Optional fifth short paragraph
Only if needed for:
- title-mismatch clarification
- mission resonance
- gap framing
- remote/location clarification

Use sparingly.

## Phase 6 — Title Mismatch and Gap Handling

These should only appear if relevant.

### Title mismatch
If the role could raise title-lineage concerns:
- address it briefly
- frame around functional overlap
- do not sound defensive
- do not over-explain

Useful pattern:
- “While my titles have not always used the same language, the work itself maps closely to…”
- “Much of my experience in [area] lived inside cross-functional systems, campaign, and enablement work rather than a formal [title] label.”

### Long-term gap
Only address if strategically necessary.

Use the calm framing from `_profile.md`:
- intentional time supporting a loved one’s health
- professional growth and renewed clarity
- never apologetic
- never melodramatic

---

## Phase 7 — Tagline Consistency

If a resume was already generated for this role, use the same role tagline so the documents feel like a matched set.

Format:
`ROLE TITLE | STRATEGIC DESCRIPTOR`

Use uppercase in the document header if the template requires it.

If no resume exists yet, generate a tagline that is:
- specific
- clean
- not hypey
- aligned to the role archetype

## Phase 8 — HTML / PDF Build

Use the cover letter template and replace all placeholders.

### Required placeholder mapping

| Placeholder | Content |
|---|---|
| `{{DOCUMENT_TITLE}}` | file name without extension |
| `{{LANG}}` | `en` unless another language is explicitly needed |
| `{{PAGE_WIDTH}}` | `7.5in` for letter, `159mm` for A4 |
| `{{NAME}}` | candidate name |
| `{{TAGLINE}}` | role tagline |
| `{{PHONE}}` | candidate phone |
| `{{EMAIL}}` | candidate email |
| `{{LINKEDIN_DISPLAY}}` | LinkedIn display text |
| `{{LOCATION}}` | candidate location |
| `{{DATE}}` | current date in `Month DD, YYYY` format |
| `{{RECIPIENT_BLOCK}}` | recipient block |
| `{{GREETING}}` | greeting line |
| `{{BODY_PARAGRAPHS}}` | body paragraphs wrapped in `<p>` tags |
| `{{SIGN_OFF}}` | sign-off selected from `writing-style.md` guidance |
| `{{TYPED_NAME}}` | candidate name |
| `{{TYPED_CONTACT}}` | concise contact line |

### Recipient block rules

If a real address is known:
```html
<div class="letter-address">
  <strong>Attn: [Name or Hiring Manager]</strong><br>
  [Company Name]<br>
  [Street Address]<br>
  [City, State ZIP]
</div>
```

If this is an online-only application:
```html
<div class="letter-address">
  <strong>Attn: Hiring Manager</strong><br>
  [Company Name]
</div>
```

Do not leave placeholder junk in the final document.

## Phase 9 — Output Paths

Use:

- Intermediate HTML: `/tmp/coverletter-{candidate}-{company}.html`
- Saved HTML: `output/cover-letters/html/coverletter-{company}-edit.html`
- PDF: `output/cover-letters/pdf/MorganEscott_{RoleTitle}_{Company}_CoverLetter.pdf`

Determine paper format:
- US/Canada → `letter`
- otherwise → `a4`

Example command:
```bash
node generate-pdf.mjs /tmp/coverletter-{candidate}-{company}.html output/cover-letters/pdf/MorganEscott_{RoleTitle}_{Company}_CoverLetter.pdf --format=letter
```
